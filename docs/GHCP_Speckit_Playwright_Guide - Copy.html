Getting Started with GitHub Copilot

Copilot is an AI pair programmer built into VS Code. It provides inline suggestions, conversational chat, autonomous agent mode, and CLI tools — all designed to accelerate your daily development workflow.

Five Ways to Interact with Copilot

💬 Chat View

Ctrl+Alt+I

Full conversations, planning, complex questions. Multi-turn dialogue with file references.

✏️ Inline Chat

Ctrl+I

Quick targeted edits directly in the editor. Select code → ask for changes.

⚡ Quick Chat

Ctrl+Shift+Alt+L

Fast one-off questions without context switching. Lightweight overlay.

👻 Inline Suggestions

Tab to accept

Ghost text completions as you type. Full method implementations, next edit suggestions.

🤖 Agent Mode

Ctrl+Shift+I

Autonomous multi-file tasks. Agent plans, implements, runs tests, and self-corrects.

Top Use Cases for Day-to-Day C#.NET Development

✅ Code Completion

Ghost text suggestions as you type. Start writing a method signature and Copilot generates the full implementation. Next Edit Suggestions (NES) predict your next edit location.

Tip: Use meaningful method names to guide better suggestions

📖 Code Explanation

Select unfamiliar code and use /explain. Ideal for understanding legacy code before migration or onboarding to a new codebase.

Prompt: "Explain how this PdfReader code handles encrypted PDFs"

🔄 Refactoring

Select code, open inline chat (Ctrl+I), describe the refactoring. Works for async conversion, pattern matching, SOLID principles, and API migration.

Prompt: "Refactor to use async/await pattern"

🧪 Test Generation

Use /tests slash command with a method selected. Generates xUnit, NUnit, or MSTest tests with edge cases, mocks, and assertions.

Prompt: "Generate xUnit tests for PdfConverter including edge cases"

📝 Documentation

Use /doc from inline chat to generate XML doc comments with <summary>, <param>, <returns>, and <exception> tags.

Tip: Select multiple methods and run /doc to batch-generate

🔍 Code Review

Select code and ask Copilot to review for null references, security vulnerabilities, SOLID violations, or performance issues. Also available in Source Control view.

Prompt: "Review for potential null reference exceptions"

Prompt Patterns That Work

Six core principles for writing effective prompts, plus patterns specifically designed for C#.NET library migration scenarios.

1. Start General, Then Specific

Give Copilot a broad description first, then list specific requirements.

// ✅ Good Write a C# method that validates an email address. Use regex, return bool, handle null input, follow RFC 5322.

2. Give Examples (Few-Shot)

Provide example input/output to guide Copilot’s understanding.

// Provide examples: ConvertPageSize(8.5, 11) => "Letter" ConvertPageSize(8.27, 11.69) => "A4" ConvertPageSize(11, 17) => "Tabloid"

3. Break Down Complex Tasks

Don’t ask Copilot to “build entire PDF pipeline”. Break it into steps.

// Step 1: Read PDF metadata // Step 2: Extract text from page // Step 3: Batch process PDFs

4. Avoid Ambiguity

Be explicit about the language, framework, and desired behavior.

// ❌ "Fix this" // ✅ "Fix the null reference // exception in ProcessDocument // when input PDF is empty"

5. Indicate Relevant Code

Open related files. Use #file, #selection, #class to attach context.

// Use chat variables: #file:PdfService.cs #selection @workspace

6. Iterate and Refine

First response not perfect? Refine. Reference previous outputs.

// After first response: "Make the method async instead" "Add error handling for empty documents"

Use Case: Library Migration with Copilot

Practical prompt patterns for migrating between libraries — demonstrated with a PDF library swap from Apryse iText to Websupergoo ABCpdf in C#.NET. These patterns apply to any library-to-library migration.

🔴 Current Stack

C#.NET with Apryse iText PDF library

Tax statement processing (Sovos), PDF generation, form manipulation, text extraction

🟢 Target Stack

C#.NET with Websupergoo ABCpdf .NET library

Same functionality, different library API — behavioral equivalence required

Migration Prompt Patterns

PhasePatternExample Prompt
1. Understand Legacy Use /explain on iText code Explain what this iText 5 code does — how it creates the PDF, adds content, and handles font embedding. I need to understand this before migrating to ABCpdf.
2. API Equivalence Ask for equivalent methods I’m migrating from iText 5 to ABCpdf 13 for .NET. What ABCpdf methods are equivalent to PdfReader.GetPageContent(), PdfStamper.GetOverContent(), and ColumnText.ShowTextAligned()? List each with a C# usage example.
3. Convert Code Select code → request conversion Convert this iText 5 C# code to use ABCpdf 13. Preserve the same functionality: reading a PDF, extracting form field values, and writing them to a Dictionary<string, string>.
4. Plan Migration Use /plan + @workspace @workspace Analyze all files that reference iTextSharp. List each file, the specific iText APIs used, and suggest an ABCpdf equivalent. Output as a migration checklist.
5. Validate Tests Generate comparison tests Generate xUnit integration tests that verify PDF generation produces the same output after migrating from iText to ABCpdf. Test page count, text content, and form field presence.
6. Document Changes Use /doc on migrated code /doc — Generate XML documentation explaining what changed from iText to ABCpdf, behavioral differences, and the ABCpdf version requirement.

Agent Mode for Migration Planning

// In Agent Mode (Ctrl+Shift+I), use this prompt: /plan Create a migration plan to replace iText with ABCpdf in this project. Consider: - Which files need changes - What the ABCpdf equivalents are for each iText API - What tests need updating - What risks exist (Unicode fonts, RTL text, large tables, embedded images) - Suggest an incremental rollout approach (one document type at a time)

Key Takeaway: Copilot excels at library migration when you break the work into phases: understand the legacy code, discover API equivalences, convert incrementally, validate with tests, and document the changes. Agent mode can automate multi-file discovery and planning.

Spec-Driven Development with spec-kit

Stop “vibe coding” with ad-hoc prompts. Spec-Driven Development (SDD) uses structured specifications as the foundation for AI-assisted code generation — producing consistent, reviewable, reproducible results.

Ad-Hoc Prompting

  • ❌ Vague, conversational requests
  • ❌ Different results every time
  • ❌ Context lost between sessions
  • ❌ No quality gates
  • ❌ Tech stack leaks into requirements
  • ❌ One-shot generation

Spec-Driven Development

  • ✅ Structured, detailed specifications
  • ✅ Consistent, reproducible output
  • ✅ Persistent spec files in repo
  • ✅ Built-in validation checklists
  • ✅ Separation of what vs how
  • ✅ Multi-step refinement

What is spec-kit?

spec-kit is GitHub’s official open-source SDD toolkit (github/spec-kit — 83,700+ stars, MIT license). It provides templates, slash commands, a CLI (specify), and extensions for structured specification-driven development. Fully supports GitHub Copilot as an AI agent and Windows/PowerShell as a first-class platform.

The SDD Workflow

1️⃣ Constitution

/speckit.constitution

Define project principles and constraints

2️⃣ Specify

/speckit.specify

Describe WHAT to build in natural language

3️⃣ Plan

/speckit.plan

Provide tech stack, get implementation plan

4️⃣ Tasks

/speckit.tasks

Break plan into actionable task list

5️⃣ Implement

/speckit.implement

Execute all tasks with AI assistance

Time Savings: Traditional vs SDD

📋 Traditional Approach (~12 hours)

  • Write a PRD (2–3 hours)
  • Create design documents (2–3 hours)
  • Set up project structure (30 minutes)
  • Write technical specs (3–4 hours)
  • Create test plans (2 hours)

🚀 SDD with spec-kit (~15 minutes)

  • /speckit.specify (5 minutes)
  • /speckit.plan (5 minutes)
  • /speckit.tasks (5 minutes)
  • ~48x faster documentation

SDD Applied to Library Migration (iText → ABCpdf)

// Step 1: Define principles /speckit.constitution This project migrates PDF generation from iText to ABCpdf in a C#.NET application. Principles: maintain behavioral equivalence, preserve existing API contracts, all unit tests must continue to pass, no rendering regression, support incremental migration (both libraries coexist during transition). // Step 2: Specify WHAT to migrate /speckit.specify Migrate the PDF generation layer from iText to ABCpdf. The system generates invoices, reports, and shipping labels. Each document type uses iText APIs for text layout, table rendering, image embedding, and barcode generation. The migration must produce visually identical output and allow incremental rollout. // Step 3: Plan HOW to implement /speckit.plan Using C#.NET 8, ABCpdf 14, xUnit for testing. Create an abstraction layer (IPdfGenerator) that both iText and ABCpdf implement. Migrate one document type at a time with feature flags. Include visual regression testing. // Step 4: Generate tasks & implement /speckit.tasks /speckit.implement

Benefits for Engineering Leadership

📐 Consistency

Every developer follows the same structured workflow. Specs are committed to the repo and reviewable in PRs. Templates enforce completeness.

🔄 Reproducibility

Same spec produces similar code regardless of developer or AI session. Specifications persist across sessions — unlike chat history.

📊 Governance

Constitution defines organizational standards. Code reviews validate against specs. Verify extension detects drift between specs and implementation.

Playwright for Automated QA

Microsoft’s open-source end-to-end testing framework. Cross-browser, auto-waiting, resilient locators, built-in codegen and trace viewer. First-class C#.NET support via NuGet — and everything works without MCP.

No MCP Required: All Playwright + Copilot features demonstrated in this session work out of the box via the Playwright NuGet package, VS Code extension, Copilot agent mode, and Copilot CLI. MCP is not needed for any core testing workflow.

Key Features

🌐 Cross-Browser

Chromium, Firefox, WebKit — one test suite covers all browsers including Edge and Safari.

⏳ Auto-Waiting

Every action auto-waits for elements to be visible, enabled, and stable. No manual waits or sleeps.

🎯 Resilient Locators

GetByRole, GetByLabel, GetByText — user-facing locators that survive UI refactors.

🎥 Codegen & Trace

Record user flows → generate C# test code. Trace viewer for step-by-step debugging with DOM snapshots.

Setup for C#.NET (MSTest)

# 1. Create a new MSTest project dotnet new mstest -n PlaywrightTests cd PlaywrightTests # 2. Install Playwright MSTest NuGet package dotnet add package Microsoft.Playwright.MSTest # 3. Build to generate playwright.ps1 dotnet build # 4. Install browsers pwsh bin/Debug/net8.0/playwright.ps1 install --with-deps

First Test (C# MSTest)

using System.Text.RegularExpressions; using Microsoft.Playwright; using Microsoft.Playwright.MSTest; namespace PlaywrightTests; [TestClass] public class ExampleTest : PageTest { [TestMethod] public async Task HasTitle() { await Page.GotoAsync("https://playwright.dev"); await Expect(Page).ToHaveTitleAsync(new Regex("Playwright")); } [TestMethod] public async Task GetStartedLink() { await Page.GotoAsync("https://playwright.dev"); await Page.GetByRole(AriaRole.Link, new() { Name = "Get started" }).ClickAsync(); await Expect(Page.GetByRole(AriaRole.Heading, new() { Name = "Installation" })).ToBeVisibleAsync(); } }

C# Locators & Assertions Cheat Sheet

User-Facing Locators (Recommended)

// Role locators (default choice) Page.GetByRole(AriaRole.Button, new() { Name = "Submit" }) Page.GetByRole(AriaRole.Link, new() { Name = "Get started" }) // Label locators (form controls) Page.GetByLabel("Email") Page.GetByLabel("Password") // Text and placeholder locators Page.GetByText("Welcome, John") Page.GetByPlaceholder("name@example.com") // Test ID locators Page.GetByTestId("submit-button")

Auto-Retrying Assertions

// Page-level await Expect(Page) .ToHaveTitleAsync(new Regex("Playwright")); await Expect(Page) .ToHaveURLAsync(new Regex("dashboard")); // Element-level await Expect(Page.GetByRole(AriaRole.Button)) .ToBeVisibleAsync(); await Expect(Page.GetByRole(AriaRole.Button)) .ToBeEnabledAsync(); await Expect(Page.GetByTestId("status")) .ToHaveTextAsync("Submitted"); await Expect(Page.GetByLabel("Email")) .ToHaveValueAsync("user@test.com");

Running Tests

dotnet test # All tests, headless dotnet test -- Playwright.BrowserName=firefox # Specific browser dotnet test -- Playwright.LaunchOptions.Headless=false # Headed mode (see browser) dotnet test -- Playwright.LaunchOptions.Channel=msedge # Microsoft Edge dotnet test --settings:.runsettings # Use config file

Copilot Agent Mode + Playwright

Agent mode is the most powerful tool for Playwright test generation. It reads your project, generates C# test classes, runs dotnet test, reads failures, and self-corrects — all autonomously, no MCP needed.

What Agent Mode Can Do (Without MCP)

📄 Read Project Context

Agent reads existing .cs test files, NuGet packages, solution structure, and .runsettings to match your coding patterns.

🏗️ Generate Complete Tests

Creates full C# test classes with test methods, locators, assertions, Page Object Models, and test initialization.

🔄 Run & Self-Correct

Executes dotnet test, reads failure output, adjusts locators and assertions, and iterates until tests pass.

Best Prompts for Agent Mode + Playwright

GoalAgent Mode Prompt
Generate test class Write a Playwright MSTest class for the login page at /login using C#. Test successful login, invalid credentials error, and empty form validation. Use GetByRole and GetByLabel locators. Extend PageTest.
Create Page Object Model Create a Page Object Model for LoginPage in C# with: email/password input locators, SignIn method, GetErrorMessage method. Constructor takes IPage. Match patterns in tests/Pages/.
Fix failing test This Playwright test fails with TimeoutException on GetByRole(AriaRole.Button, “Submit”). The button is inside a modal dialog that appears after clicking “Add Item”. Fix the test to wait for the dialog first.
Generate from patterns Look at the existing Playwright tests in Tests/E2E/ and generate similar tests for the /settings page. Follow the same C# patterns for locators, assertions, and organization.
Add edge cases Based on the TodoPage page object, generate additional MSTest methods for: long text (255+ chars), special characters, empty input, and toggling all items.

Example: Page Object Model Pattern (C#)

using Microsoft.Playwright; namespace PlaywrightTests.Pages; public class LoginPage { private readonly IPage _page; public LoginPage(IPage page) => _page = page; // Locators private ILocator EmailInput => _page.GetByLabel("Email"); private ILocator PasswordInput => _page.GetByLabel("Password"); private ILocator SignInButton => _page.GetByRole(AriaRole.Button, new() { Name = "Sign In" }); private ILocator ErrorMessage => _page.GetByRole(AriaRole.Alert); // Actions public async Task GotoAsync() => await _page.GotoAsync("/login"); public async Task LoginAsync(string email, string password) { await EmailInput.FillAsync(email); await PasswordInput.FillAsync(password); await SignInButton.ClickAsync(); } // Assertions public async Task ExpectErrorMessageAsync(string text) => await Assertions.Expect(ErrorMessage).ToContainTextAsync(text); } // Usage in tests: [TestClass] public class LoginTests : PageTest { private LoginPage _loginPage = null!; [TestInitialize] public async Task Setup() { _loginPage = new LoginPage(Page); await _loginPage.GotoAsync(); } [TestMethod] public async Task ValidLogin_NavigatesToDashboard() { await _loginPage.LoginAsync("admin@example.com", "password123"); await Expect(Page).ToHaveURLAsync(new Regex("/dashboard")); } [TestMethod] public async Task InvalidLogin_ShowsError() { await _loginPage.LoginAsync("admin@example.com", "wrong"); await _loginPage.ExpectErrorMessageAsync("Invalid credentials"); } }

Copilot CLI for Playwright

GitHub Copilot CLI (copilot command) is a standalone AI agent for the terminal. Interactive mode starts a conversation; Shift+Tab switches to Plan mode for structured analysis. It can set up Playwright, run tests, debug failures, and create CI workflows — all from the command line.

# Interactive mode — start a conversation copilot > Set up Playwright in this .NET project with MSTest. Install the NuGet package, build, install browsers, and create a first test for the homepage. # Run and fix tests copilot > Run the Playwright tests with dotnet test and fix any failures you find # Create CI workflow (single prompt) copilot -p "Create a GitHub Actions workflow that runs .NET Playwright tests on push to main. Include dotnet build, browser install, dotnet test, and upload playwright trace artifacts." --allow-tool='shell(dotnet)'

Test Authoring Workflow with Copilot

A recommended end-to-end workflow that combines recording, AI generation, refinement, and CI/CD — using the tools available without MCP.

1

Record Baseline with Codegen

Use VS Code extension (“Record new”) or CLI (pwsh playwright.ps1 codegen) to record user flows. Generates C# test code with best locators.

2

Refine with Copilot Agent Mode

Open generated test, ask Copilot to replace CSS selectors with GetByRole/GetByLabel, add meaningful assertions, add test cleanup.

3

Extract Page Object Models

Ask agent mode to extract locators and actions into Page Object classes. Keeps tests clean and maintainable.

4

Generate Edge Cases & Negative Tests

Use agent mode to generate additional test methods: boundary values, special characters, empty inputs, error flows, accessibility checks.

5

Create CI/CD Pipeline

Use Copilot CLI or agent mode to generate GitHub Actions workflow with dotnet build, browser install, dotnet test, and trace artifact upload.

6

Debug with Trace Viewer

Analyze failures with step-by-step trace viewer. DOM snapshots, network monitoring, console logs, and source mapping. Use “Fix with AI” sparkle icon for Copilot-assisted fixes.

Next Steps & Getting Started Checklist

  • Share prompt patterns — create .prompt.md files for common team tasks
  • Review Copilot adoption metrics and identify areas for deeper training
